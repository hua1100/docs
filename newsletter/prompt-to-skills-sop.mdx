## 還只會寫 Prompt 嗎？

AI 能做的事情，已經不限於你與它的聊天室之內了。

想像一下，如果你收到一個英翻中的翻譯需求，繳交格式要求是一個 Word 文件，文件中有四行列表，分別是翻譯編號、翻譯進度、原文、譯文，翻譯後你要將譯文用**追蹤修訂**的方式填進表格中，並寄給主管。你會選擇哪種處理方式？

---

## 五個等級的自動化

<Steps>
  <Step title="Level 1：單次對話">
    貼上原文，每次輸入「幫我翻譯」，AI 翻譯，自己把翻譯一行行填入表格，寄出信件。
  </Step>
  <Step title="Level 2：指令範本化">
    運用思維鏈（CoT）等 Prompt 技巧，建立斜線指令，AI 翻譯，自己把翻譯一格一格填入表格，寄出信件。
  </Step>
  <Step title="Level 3：知識庫加持">
    建立專案，知識庫中放入翻譯常用詞彙對照表，貼上原文用斜線指令，AI 翻譯，自己把翻譯一格一格填入表格，寄出信件。
  </Step>
  <Step title="Level 4：流程腳本化">
    在專案中使用斜線指令翻譯，運用 Skills 建立 SOP 讓 AI 把譯文貼到文件中，下載檔案、寄出信件。
  </Step>
  <Step title="Level 5：系統整合">
    在專案中使用斜線指令翻譯，運用 Skills 建立 SOP 讓 AI 把譯文貼到文件中，連接 MCP 把完成的翻譯檔案自動給主管。
  </Step>
</Steps>

## 工具一張表秒懂

| 項目 | 臨時指令（Prompts） | 標準流程（Skills） | 專案空間（Projects） | 外部連接（MCP） |
|------|---------------|--------------|----------------|------------|
| **這是什麼？** | 現在馬上跟 Claude 說要做什麼 | 做事的固定步驟，可以重複使用 | 長期專案的資料庫和工作區 | 讓 Claude 連到其他軟體/工具 |
| **什麼時候用？** | 臨時需求、一次性的事 | 常做的事、有固定做法的工作 | 長期專案、需要團隊共用資料 | 要從 Google Drive、資料庫抓資料 |
| **會記得多久？** | 只有這次對話，結束就忘 | 一直都在，隨時可用 | 在專案裡一直記得 | 只要設定好就一直可用 |
| **一句話說明** | 「我現在要你做這個」 | 「照這個步驟做」 | 「專案的所有資料在這」 | 「去別的地方抓資料來用」 |

## 製作 Skills 的四個避雷指南

### 1. 不要用預設的 skill-creator

一個 Skill 可以很簡單，只有一個 markdown 文件；也可以非常複雜，由多個腳本組成，甚至透過腳本呼叫 MCP。

<Warning>
我一開始很相信 Claude Sonnet 4.5 的能力，用 web 版 Claude 的內建 skill-creator 製作，做完也沒有先在本地測試腳本能不能跑，就直接安裝到 Claude 中，結果就是在執行過程中 debug 直接燒光我的 Token，只花四天逼近我的 weekly limit。
</Warning>

在 AI 小聚讀書會碰到 YC，他直接說一定要用有 agent 功能的 claude code 來做，網頁版 claude 沒有推理反思的功能，能力落差 claude code 一大截。

後來的做法是，把官方的最佳實踐作為 claude code 的系統提示詞檔案，研究最好的流程，再分別在本地測試，確定能跑才上 web claude。

### 2. SOP 也要 AI Friendly 版

Skills 就是一套定義好的 SOP，這套給人類看的 SOP 再轉換成 Skills 時，也要考慮到 AI 的能力邊界，微調成 AI Friendly 版。

<Tip>
OCR 不只用於把紙本變成電子檔，也可以把 PDF、Word 轉換成 AI 好讀版 MD 檔，讓 AI 辨識表格內容事半功倍。推薦工具：MarkItDown、PaddleOCR、MinerU、DeepSeek-OCR。
</Tip>

### 3. 一個 Skill 可以有多少個工作步驟？何時要拆分？

若在 Skill 中想發揮「顯示思考流程」的威力，請注意：思考鏈本身就已經佔用大量 token，當後面還有 5、6 個編輯流程要進行，AI 容易忘記前面的步驟，造成流程中斷，因此建議拆成兩個階段來處理。

最適合的步驟與流程還是要透過 evaluation（評估）才知道，官方的最佳實踐中也有包含評估指引，可以餵給 AI 請它依照步驟幫你評估。

### 4. 在 SKILL.md 中明確規定「不能做哪些事」

在測試過程中，Claude 不會每次都遵照說明文件中的指令，而是自己判斷這個狀況應該使用什麼腳本，自己從頭生成。除了說明文件和腳本要優化之外，還可以在 SKILL.md 的 description 處加上禁止事項，例如「僅使用以下腳本，當腳本出現錯誤應停止流程並告知使用者。」讓 AI 能主動回報。
